Graph 4 of the series. How vit explain traces a Δ back to logged sets — two lanes for its two modes, shared curve primitives drawn once in the middle.

Reading the graph. vit explain has two modes, drawn as two lanes: the exercise lane (top) traces the per-exercise Δ behind vit progress / vit curve --diff; the cluster lane (bottom) traces the cluster-level Δ behind vit progress --by muscle / vit review. Both lanes call into the same four shared primitives from queries/curve.py, drawn once in the middle band — dark arrows are data / lane flow (left → right), thin grey arrows are function calls into the primitives, and the two small arcs inside the band show that curve() itself composes observed_points + estimate_1rm.

Auditability — the command's reason to exist. Every number in either trace resolves to a set row in a workout file: each observation line prints date · reps @ weight · markers, which is exactly one set in programming/workouts_v2/YYYY-MM-DD.wo.yaml; each anchor (★), formula application, laterality factor and projection coefficient is printed on the line that used it, so the final Δ can be followed backwards, term by term, to logged sets.

Fidelity notes. Both lanes route anchor selection through curve()'s recency-decayed winner (rm1_point) — the same pick curve_diff / vit progress use. The exercise lane calls curve() once per window in _explain_exercise, and _print_window_obs marks that winner (★) while still printing every observation's raw estimate_1rm for the audit trail; the cluster lane's direct branch does the same via _pick_with_explain, mirroring _window_rm1. Projected observations never enter when direct data exists; with no direct data the anchor is the median projected 1RM.
