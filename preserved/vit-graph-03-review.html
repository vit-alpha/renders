Graph 3 of the series. The most-composed command: three mode lanes over a shared query layer.

Reading the graph. One command, three mode lanes. cmd_review dispatches on the arguments — each lane’s pale-blue chip is that dispatch entering the lane (bare → overview; cluster id, which wins over an exercise id → cluster drill; exercise id → exercise). The query layer column is shared: cluster_progress feeds lanes 1 and 2, curve_diff feeds lane 2 and (via review_exercise) lane 3, session_context feeds lanes 2 and 3. Lane 3 has a second stage: --session DATE short-circuits _exercise into _exercise_session without calling review_exercise at all.

repo area · data · tools (queries & command modes) · primitives / dispatch chips (lighter border) · outputs. Nothing in this graph lives outside the repo.

The Δ numbers. Every displayed 1RM anchor is curve()’s rm1_point — the recency-decayed winning observation — so the anchor brackets, laterality factors and shape-mismatch flags all describe the same set the Δ was computed from. The overview’s base Δ column comes exclusively from cluster_progress (direct observations win; graph-projected observations fill gaps); review_overview also computes a per-top-exercise Δ internally, but the CLI never displays it.

Sessions and reliability. Sessions are identified by workout file (session_key), so split days like 2026-06-28_3pm / _6pm count separately in every table. --min-reliability gates workouts in all three modes (volume, tables, session lists) — but note it is not threaded into the Δ path (cluster_progress / curve_diff / observed_points take no reliability argument), nor into the --session drill.
